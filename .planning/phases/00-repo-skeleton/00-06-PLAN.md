**Create `docs/MIGRATIONS.md`** — migration discipline (INFRA-08). Required content:

  1. **Goal** — Schema changes must not lock production tables for more than a few hundred milliseconds.
  2. **Rule 1: Nullable-first column adds** — new columns are added as `NULL` first, backfilled in a separate migration, then `ALTER COLUMN ... SET NOT NULL` in a third migration. Never `ADD COLUMN ... NOT NULL DEFAULT 'x'` on a populated table (rewrites every row, holds AccessExclusiveLock).
  3. **Rule 2: `CREATE INDEX CONCURRENTLY`** — for index adds on populated tables. Concurrent index creation does not block writes. NOTE: `CREATE INDEX CONCURRENTLY` cannot run inside a transaction; the migration must use Alembic's `op.execute(...)` with appropriate transaction boundary handling (Phase 1+ will land an example).
  4. **Rule 3: Constraints as `NOT VALID` then `VALIDATE`** — when adding a CHECK or FK to a populated table, first `ALTER TABLE ... ADD CONSTRAINT ... NOT VALID`, then `ALTER TABLE ... VALIDATE CONSTRAINT ...` in a separate migration (NOT VALID skips the full-table check; VALIDATE acquires only a SHARE UPDATE EXCLUSIVE lock).
  5. **Rule 4: Drop columns are two-phase** — first ship a migration that removes all code references; later ship the actual `ALTER TABLE ... DROP COLUMN`. This prevents downtime if a rollback is needed.
  6. **Rule 5: Foreign-key adds use `ON DELETE CASCADE` carefully** — household_id FKs use CASCADE so deleting a household cleans children; intra-domain FKs use `ON DELETE RESTRICT` by default (prevents accidental orphan-deletes).
  7. **RLS convention (Phase 0 reservation; Phase 1 activation)** —
       - `ALTER TABLE <name> ENABLE ROW LEVEL SECURITY` per domain table as it's created.
       - Policy template: `CREATE POLICY <name>_household_isolation ON <name> USING (household_id = current_setting('app.current_household_id', true)::uuid)`.
       - **The `, true` second argument to `current_setting` is non-negotiable** — without it, an unset GUC raises `unrecognized configuration parameter`. Per RESEARCH.md Pitfall 9 (lines 1132-1138).
       - The api's `get_session` will issue `SET LOCAL app.current_household_id = '<uuid>'` after auth resolves the household (Phase 1).
       - GUC is reserved in Phase 0 via the `COMMENT ON DATABASE` in the 0001 baseline migration (no policies; convention only).
  8. **Migration naming** — `NNNN_short_description.py` (e.g., `0001_phase0_baseline.py`, `0002_phase1_household.py`). The revision identifier inside the file matches the filename prefix.
  9. **Running migrations** — `make migrate` (which invokes `docker compose exec api alembic -c packages/db/alembic.ini upgrade head`).

**Create `docs/adr/001-modular-monolith.md`** — Architecture Decision Record. Standard ADR format:

  - **Status**: Accepted (2026-05-23)
  - **Context**: PRANAV is a self-hostable personal finance app. Single-developer team. Need fast iteration, low operational complexity, and a clear path to scale if the cloud version ever needs to handle thousands of households.
  - **Decision**: Modular monolith. One deployable (`apps/api/`); modules live under `apps/api/src/modules/<name>/` with hard import boundaries enforced by convention (Phase 0) and `import-linter` (Phase 1+). Cross-module calls flow through `packages/domain/`. The web app (`apps/web/`) is a separate process (Next.js requires its own runtime) but shares types via `packages/shared/`.
  - **Consequences (positive)**: Single deploy unit means one Dockerfile, one CI pipeline, one process to monitor. Refactoring across modules is easy (single repo, single language per layer). RLS provides multi-tenancy without needing per-tenant databases. Future split into microservices is cheap because boundaries are already enforced.
  - **Consequences (negative)**: Cannot scale individual modules independently. A single bad query can block the whole process (mitigated by async + Postgres connection pool). Local dev runs everything in containers, requiring ~2GB RAM (acceptable for the target audience).
  - **Alternatives considered**: Microservices (rejected — overkill for solo dev + initial scale; per-service Postgres + service mesh is enterprise complexity). Single Django/Rails monolith (rejected — async-first FastAPI fits the forecast endpoint's chatty nature better; CLAUDE.md mandates async SQLAlchemy). Phase-1-immediate-multi-tenant (rejected — RLS scaffolding in Phase 0 buys correctness with no compromise).
  - **Source documents**: CLAUDE.md, .planning/research/ARCHITECTURE.md, RESEARCH.md.
